Web

介绍

Web是什么

Web即全球广域网

也称万维网(www -> World Wide Web)

就是能通过浏览器访问的网站

工作流程

Web网站工作流程(前后端分离模式)↓↓↓

  • 组成内容

    前端服务器(前端程序)

    后端服务器(Java程序)

    数据库服务器

  • 工作流程

    1. 用户在浏览器通过域名向前端服务器(对应前端程序)发起请求,前端服务器响应返回前端程序代码(即展示样式),浏览器解析前端代码展示效果
    2. 前端服务器同时也会返回一个地址,表示获取数据的地址.则浏览器向后端服务器发起请求,后端服务器继续向数据库服务器发起请求
    3. 数据库服务器响应,返回数据至后端服务器,然后后端服务器响应将数据返回给浏览器(即展示数据)

  • Web网站开发模式

    前后端分离 -> 前端工程师开发前端程序,后端工程师开发后端Java程序

    混合开发(前后端不分离) -> 浏览器发起请求后,将前端程序与数据一并返回给浏览器

浏览器

前端代码经过浏览器的解析和渲染转化成用户看到的网页页面

浏览器内核

  • 即浏览器中对代码进行解析渲染的部分
  • 不同浏览器因为内核不同,解析效果会有差异

需要制定统一标准↓↓↓

Web标准

也称为网页标准

  • CSS : 负责网页的表现(页面元素的外观样式)

同步与异步的请求方式

同步 -> 客户端向服务器发起请求, 服务器需要时间做出响应, 与此同时客户端只能等待服务器的响应, 服务器响应完毕将数据返回客户端, 客户端重新加载完毕

异步 -> 客户端向服务器发起请求, 服务器需要时间做出响应, 与此同时客户端还可以执行其他操作, 不需要一直等待服务器的响应, 服务器响应完毕将数据返回客户端, 客户端重新加载完毕

前端路由Router

前端路由

介绍

  • 前端路由是指在前端应用程序中管理页面导航和 URL 的机制

作用

  • 前端路由通过监听 URL 的变化，并根据 URL 的路径来决定加载哪个组件或页面，从而实现在用户操作时动态地更新页面内容，而不需要重新加载整个页面

一句话概括

  • 前端路由通过识别网页URL路径和组件(页面)的对应关系, 动态地更新页面, 而不用刷新整个页面

分类

常用前端路由库有

  • React Router
  • Vue Router
  • Angular Router

Vue Router

介绍

  • Vue Router是Vue官方路由

组成

  • VueRouter 路由器类

    在VueRouter类中存放了路由表, 表中记录了url的hash(#)路径和组件(.vue文件)之间的对应关系

    比如 url/user对应UserView.vue组件, url/emp对应EmpView.vue组件…

  • <router-link> 请求链接组件

    最终被浏览器解析为<a>超链接标签

    可以在router-link请求链接组件中声明该链接访问的是哪一个vue组件

    比如<router-link to="/emp">员工管理</router-link>

  • <router-view> 动态视图组件

    用来渲染和展示与路由路径对应的组件

    想要在哪个组件展示, 就在该组件中加入<router-view></router-view> 标签

安装

npm instal vue-router@3.5.1

但是在配置Vue项目时可以直接勾选Router路由文件, 一起加载到项目中, 就不用再安装了

定义路由

  • 在src目录下的router目录中的index.js文件中定义路由配置信息

设置path 和 name 以及 component路径 三者的对应关系

  • path -> 指定url路径名( /#/example )
  • name -> 指定路由的唯一标识符
  • component路径 -> 指定访问的vue组件

一般还要配置根路径/ ,否则运行项目打开是一片空白

  • redirect -> 重定向, 即访问/根目录时, 会自动定向到某一个页面

    如{path: '/', redirect:'/dept'} 加载根目录时直接跳到部门管理界面

使用路由

默认已经将router目录导入main.js入口js文件中了, 所以修改完router/index.js路由信息后就能立刻使用~

HTTP协议

介绍

HTTP

概念

  • Hyper Text Transfer Protocol, 超文本传输协议, 规定了浏览器与服务器之间传输数据的规则

这个传输数据的规则怎么理解呢?

  • 浏览器向服务器请求数据的时候, 要给它发一段话(即请求数据字符串), 表明我要什么, 你给我返回的格式是怎样等等
  • 服务器向浏览器响应数据的时候, 也要给它发一段话(即响应数据字符串), 表明数据是什么, 你要如何解析等等

这两段传输数据的过程, 都利用了HTTP协议进行数据的规范化传输

所以一句话概括

=> HTTP就是传输数据的格式规则, 一方面是请求数据传输规则, 一方面是响应数据传输规则

特点

  • 基于TCP协议 : 面向连接, 更加安全

  • 基于请求-响应模型

    一次请求对应一次响应, 没有请求就没有响应

  • HTTP协议是无状态的协议

    即对事务处理没有记忆能力, 每一次请求-响应都是独立的

    缺点 -> 多次请求间不能共享数据

    优点 -> 速度快

对于这个”无状态”协议, 可以用一个例子理解一下

比如我现在登录了一个教育网站, 由于我输入了正确的账号密码, 我有资格访问我的账户数据了

但是接下来我点击网站里面的其他功能, 就比如课程管理栏目, 正常来说由于我登录了我的账号, 我有一个登录的信息在此, 我有资格访问我能访问到的数据; 可是由于HTTP的无状态协议, 我这一次对课程管理数据的访问和上一次我的登录数据之间没有关系, 在我想要访问课程数据的时候, 服务器不知道”我是谁了”(因为上一次的登录数据响应之后已经结束了), 那我到底有没有访问课程数据的资格呢? -不知道

GET/POST请求方式区别

GET请求方式 -> 请求参数在请求行中(在路径中通过?key=value形式传递), 没有请求体, 且有请求大小限制

​ 如: /01.GET-POST.html?name=Tom&password=123

POST请求方式 -> 请求参数在请求体中(不会将请求参数写在网页上方url路径中), POST请求方式的大小无限制

响应状态码分类

一般是100+ ~ 500+

响应状态码 含义
1xx 响应中-临时状态码, 表示客户端请求已经被服务器端接收, 告诉客户端应该继续发送请求或者已完成就忽略它(比较少用)
2xx 成功-表示请求已经被成功接收, 处理已完成
3xx 重定向-重定向到其他地方(浏览器自动完成), 让客户端再发起一次请求以完成整个处理
4xx 客户端错误-处理发生错误, 责任在客户端 如请求了不存在的资源/客户端没有被授权/禁止访问等
5xx 服务器错误-处理发生错误, 责任在服务器 如程序抛出异常了

常见响应状态码

状态码大全 -> 状态码文档

状态码 英文描述 解释
==200== OK 客户端请求成功，即处理成功，这是我们最想看到的状态码
302 Found 指示所请求的资源已移动到由Location响应头给定的 URL，浏览器会自动重新访问到这个页面
304 Not Modified 告诉客户端，你请求的资源至上次取得后，服务端并未更改，你直接用你本地缓存吧。隐式重定向
400 Bad Request 客户端请求有语法错误，不能被服务器所理解
403 Forbidden 服务器收到请求，但是拒绝提供服务，比如：没有权限访问相关资源
==404== Not Found 请求资源不存在，一般是URL输入有误，或者网站资源被删除了
405 Method Not Allowed 请求方式有误，比如应该用GET请求方式的资源，用了POST
428 Precondition Required 服务器要求有条件的请求，告诉客户端要想访问该资源，必须携带特定的请求头
429 Too Many Requests 指示用户在给定时间内发送了太多请求（“限速”），配合 Retry-After(多长时间后可以请求)响应头一起使用
431 Request Header Fields Too Large 请求头太大，服务器不愿意处理请求，因为它的头部字段太大。请求可以在减少请求头域的大小后重新提交。
==500== Internal Server Error 服务器发生不可预期的错误。服务器出异常了，赶紧看日志去吧
503 Service Unavailable 服务器尚未准备好处理请求，服务器刚刚启动，还未初始化好

Tomcat服务器执行流程

在B/S架构中, 浏览器需要和服务器进行数据的请求和响应

当浏览器向服务器发起请求时, 就访问到部署在Tomcat中的应用程序, 然后各种Controller类接收请求后响应对应结果, 中间数据传输遵循HTTP协议

然而, 之前所写的controller包下的HelloController类案例, 只是一个简单的Java类, 并没有实现各种接口, 那么Tomcat就无法识别, 但是Tomcat会识别一个程序DispatcherServlet(后面简称DS), 它就实现了Servlet规范中的Servlet接口, 使得Tomcat能够识别

DispatcherServlet继承了Servlet接口, 所以Tomcat可以识别这个程序, 当浏览器向服务器发起请求时, 会先经过DS这个程序, 它将任务分发给各种Controller程序, 然后响应完数据再发送给DS, 然后再响应给浏览器

由于DS的重要性, 也被称为核心/前端控制器 其包括两个部分

  • HttpServletRequest请求对象 -> 获取请求数据

    当Tomcat服务器接收到浏览器发起的请求时, 会经过DS程序将请求内容解析封装成一个请求类对象

  • HttpServletResponse响应对象 -> 设置响应数据

    当Tomcat服务器解析完所有请求信息时, 会将响应的数据封装成一个响应类对象, 在其中设置响应数据并响应给浏览器

一句话理解Tomcat服务器执行流程

=> 浏览器发起请求信息被SpringBoot内置的DS程序封装为请求对象, 然后被各种Controller程序执行, 返回的响应信息被DS程序封装为响应对象, 返回给浏览器

SpringBootWeb分层与解耦

三层架构

单一职责原则

单一职责原则

  • 在软件开发的时候, 尽量让每一个接口/类/方法职责更加单一, 只做一件事情
  • 作用 -> 降低程序复杂度, 提高可读性, 拓展性, 可维护性

基于单一职责原则, Web开发中出现”三层架构”的概念

三层架构含义

三层架构

  • Controller -> 控制层, 负责接收前端发起的请求, 并响应数据
  • Service -> 业务逻辑层, 处理具体业务逻辑
  • Dao(Data Access Object) -> 数据访问层/持久层, 负责数据的访问操作, 包括数据的增删改查

三层架构处理请求的流程

  1. 前端(浏览器)发起请求
  2. Controller 层接收请求, 将不同的任务分发给不同 Service 方法
  3. Service层执行业务逻辑, 需要调用Dao层来执行数据的访问操作
  4. Dao层访问实际数据, 可能是数据库或者文件系统等, 执行相应的增删改查操作, 将操作结果返回给Service层
  5. Service层对Dao层返回的数据结果进行进一步处理, 然后返回给Controller层
  6. Controller层将数据结果响应前端(浏览器), 通常是JSON,HTML等格式

=> 所以, 当我们要修改业务逻辑的时候, 不用修改Controller层和Dao层, 只要更改Service层即可, 利用三层架构满足了单一职责原则~

分层解耦

内聚与耦合

内聚

  • 软件中各个功能模块内部的功能联系

    比如员工管理的Service逻辑处理中, 只包含和员工相关的逻辑处理功能, 那么其内聚就高

耦合

  • 衡量软件中各个层/模块之间的依赖, 关联程度

    比如三层架构中, Controller层紧密依赖Service层处理的结果, 而Service层由紧密依赖Dao层的数据访问, 所以CS之间和SD之间耦合就高, 就不好

软件设计要满足的原则是 高内聚, 低耦合

解除耦合

以Controller层与Service层之间的依赖关系说明为例 ↓

当我们想要尽量降低他们之间的耦合度的时候, 会发现Controller层一直要接收Service层处理的数据对象, 那么每当Service层变动时, 传递给Controller层的数据对象就会变动, 那么这个万恶的数据对象就将他们二者强制性地连接在一起了

为了降低/解除C和S的耦合度, 就需要对这个可恶的数据对象进行特殊处理, 保证当Service层变动的时候, Controller接收的数据对象也不会改变, 这样就不会有强制连接的高耦合度了

在此, 我们可以使用一个容器来存储Service层处理的数据对象, 供Controller层访问Service层处理的数据. 即使Service层变动了, 其处理创建的数据对象也保存在这个容器中, 让Controller层直接访问这个容器就可以获取数据对象了, 避免与Service层的强耦合

这个能解除/降低多层之间的耦合度的容器, 就是我们亲爱的IOC容器(或称Spring容器)

在这个容器中所创建, 管理的对象, 就称之为 bean对象

但是, 此处还存在两个问题

  • Service层如何把处理好的数据对象存到这个容器 ?

    解决方法 -> 控制反转(IOC)

    指的是 对象的创建控制权, 由程序本身转移到外部容器

  • Controller层如何访问存储在容器中的已经处理好的数据对象 ?

    解决方法 -> 依赖注入(DI)

    指的是 容器会为应用程序提供其运行时所依赖的资源

IOC&DI流程

解耦合流程

  1. 控制反转

    将Service层和Dao层的实现类, 交给IOC容器管理

    在Service和Dao的实现类上方添加 @Component 注解, 令这个类成为IOC容器中的Bean对象

  2. 依赖注入

    为Controller层和Service层注入运行时, 所依赖的对象

    在Controller和Service方法中, 数据对象(成员变量)上方添加 @Autowired 注解, 使IOC会提供该类型的Bean对象, 并为该对象赋值

  3. 运行测试即可

IOC详解

控制反转的含义

IOC -> Inversion of Control, 控制反转

作用 => 会将对象的创建权交给IOC容器, 在IOC容器中创建/管理这些bean对象, 只要在对应的类上方添加 @Component 注解, 就表示将这个类的对象声明为bean对象

Component衍生注解*3

在Spring框架中, 除了Component注解以外, 为了更好地标识Web应用程序中的bean对象的归属层, 还提供了Component的三个 衍生注解

  • @Controller

    标识Controller层的bean对象, 但是由于@RestController注解包含了@Controller注解和@ResponseBody注解, 只需要写一个 @RestController 注解即可

  • @Service

    标识Service层的bean对象

  • @Repository

    标识Dao层的bean对象

    不过这个注解很少用, 之后会被Mybatis相关的注解替代

@Component注解可以为 “不属于三层架构中, 但又想交给IOC容器管理的类”声明bean对象, 比如工具类

  • 默认的扫描范围就是启动类所在的包, 以及子包

DI详解

依赖注入的含义

DI -> Dependency Injection, 依赖注入

作用 => IOC容器要为程序提供运行时, 所需要/依赖的资源, 这个资源其实就是bean对象, 在此步骤中要使用 @Autowired自动装配注解, 默认按照bean对象的类型进行装配, 即在IOC容器中寻找这个类型的bean对象, 然后注入程序

但是, 当一个程序中有重复多个同一对象类型bean对象存在时, 运行程序时就会出现自动装配错误

Bean注入

由于 @Autowired自动装配 注解默认按照bean对象的类型进行程序的注入, 当存在多个相同类型的bean对象时, 就会出错

解决方法

  • @Primary 注解

    设置bean对象的优先级, 想让谁生效, 就在谁的上面加上这个注解

    例如当EmpServiceA与ESB的Service类型bean对象冲突时, 为A添加Primary注解, 提高它的bean对象优先级, 则优先使用这个bean对象

  • @Qualifier 注解

    在使用到这个bean对象的程序中, 写入@Qualifier(value="bean名称")注解, 指定使用的bean对象(当然value关键字可省略)

    在下面这个demo中, 就指定了empService使用的是empServiceA这个bean对象

  • @Resource 注解

    按照bean对象的名称进行注入, 此注解由JDK提供

    在使用到这个bean对象的程序中, 可以删去@Autowired注解, 只要写入 Resource(name=”bean名称”)注解即可, 指定使用的bean对象

    注意@Qualifier注解和@Resource注解的使用区别~

@Resource 与 @Autowired 这两个注解的区别

  • @Autowired

    是Spring框架提供的注解

    默认按照bean对象类型进行注入

  • @Resource

    是JDK提供的注解

    默认按照bean对象名称进行注入

技术介绍

在之前的学习中, 我们都是利用图形化界面工具连接数据库管理系统后, 对数据库进行操作. 这是Dao层(数据访问层)中数据访问的数据库模式中的一部分

但是, 访问数据库还有一种方法就是通过Java程序, 而不是DBMS, 现在主流的通过Java程序访问数据库的技术就是 Mybatis

什么是Mybatis

  • 一款持久层/数据访问层(即Dao层)框架, 用于简化JDBC的开发

    JDBC -> Java DataBase Connection, JavaEE13项规范之一, 是操作数据库的规范

    使用JDBC进行开发过于繁琐, 所以现在多用Mybatis这种封装JDBC的框架进行实际开发, 提升效率

JDBC介绍

是什么

Java DataBase connectivity, 使用 Java语言操作关系型数据库 的一套API

son公司只提供jdbc的接口规范, 实际的接口内容都是各个RDBMS厂商自行编写的, 这个jdbc的接口实现, 就是驱动的jar包

代码..不用掌握

原始jdbc程序存在的问题

  • 硬编码, 即写死了的Java代码 (如String url = “jdbc:mysql://localhost:3306/mybatis”;)

    后续想要修改就很麻烦 -> 重新编译, 打包, 运行

  • 解析结果繁琐

    一个一个字段解析, 太繁琐

  • 资源浪费, 性能降低

Mybatis中如何解决这些问题

  • Mybatis中将数据库连接四要素配置在配置文件中
  • 不用解析结果, 自动将数据库返回的结果封装到对象中
  • 使用 数据库连接池 , 避免频繁的连接与断开造成的资源浪费

数据库连接池

没有数据库连接池时, 程序使用SQL语句访问数据库的流程

  1. 客户端(程序)要先创建一个连接对象
  2. 连接对象使用SQL语句访问数据库
  3. 销毁连接对象释放资源

是什么

  • 一个负责分配, 管理数据库连接对象的容器

作用

​ 在有了数据库连接池之后, 程序使用SQL语句访问数据库的流程如下

  1. 程序启动时, 就会先在数据库连接池这个容器中, 初始化若干连接对象
  2. 客户端从连接池中获取一个连接对象, 执行SQL语句访问数据库
  3. 客户端归还连接对象, 返回连接池中

  • 数据库连接池实现了连接对象的复用, 而不是创建新的连接然后又销毁

  • 数据库连接池会自动判断连接对象的空闲时间, 并且自动回收

    比如说, 连接池中某一个连接对象被一个程序进程(客户端)借走了, 但是这个客户端没有执行SQL语句, 让连接对象一直闲置在那里不归还数据库连接池, 那么数据库连接池会对其空闲时间进行记录, 当空闲时间达到最大空闲时间时, 数据库连接池会强制回收该连接对象, 以避免连接池中其他客户端借用连接对象的遗漏(即连接对象被借完了还没还, 下一个客户端没法使用连接对象)

连接池的优势

  • 资源重用

    即数据库的连接对象不会一次性地使用, 而是反复利用

  • 提升系统响应速度

  • 避免数据库连接遗漏

数据库连接池的实现

要使用sun公司的标准数据库连接池接口 DataSource , 然后由各个第三方自行实现接口的实现类

该接口的一个核心功能 getConnection() 方法, 是用来获取连接对象的, 调用该方法就能从数据库连接池中获取一个连接对象

数据库连接池并不需要我们自行实现, 使用现成的数据库连接池技术即可, 如

  • Druid德鲁伊

    阿里巴巴开源项目, 是Java语言的最好数据库连接池之一

  • Hikari追光者(SpringBoot默认自带连接池)

  • C3P0和DBCP已经过时了~

Lombok

是什么

Lombok是一个Java类库, 通过注解的形式, 自动为实体类生成包括但不限于 构造器, getter/setter, equals, hashcode, toString等方法 , 还可以生成自动化生成日志变量

简化Java开发, 提高效率

原理

在程序编译的时候, 根据注解的内容, 为实体类的属性提供对应的方法, 而且对于static修饰的属性, 会自动忽略equals和hashCode方法以及AllArgsConstructor方法中的静态字段部分参数

为什么要用lombok? =>

在之前我们编写JavaBean类(事物类)的时候, 写完成员变量时总要再编写各种成员方法, 即使不用自己手写, 采用自动生成的方式, 类中的代码看起来还是很臃肿

此时, 可以利用lombok这个Java类库解决这个问题

如何使用lombok? => 学会常用注解即可

注解 作用
@Getter/Setter 为实体类的所有属性提供get/set方法
@ToString 为实体类生成便于阅读的toString方法
@EqualsAndHashCode 为实体类中非静态字段自动重写equals方法和hashCode方法
==@Data== 为实体类提供前四个注解包含的内容, 即@Getter+@Setter+@ToString+@EqualsAndHashCode
@NoArgsConstructor 为实体类生成无参的构造方法
@AllArgsConstructor 为实体类生成非静态字段的带有各参数的构造方法
